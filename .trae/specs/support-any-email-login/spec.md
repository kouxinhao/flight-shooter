# 支持任意邮箱登录

## 为什么

目前 QQ 邮箱 SMTP（2810991744@qq.com）已配置，但用户反映只有该邮箱能收到验证码。需要确认任意邮箱都能正常登录，并在 SMTP 不可靠时提供降级方案。

## 调查结论

**登录系统本身已支持任意邮箱**— `/send-code` 和 `/verify-code` 接口不限制邮箱域名，任何有效邮箱地址都可以注册/登录。但 QQ 免费邮箱 SMTP 存在以下限制：

- QQ 免费邮箱 SMTP 有每日发信限额（约 50-100 封）
- 某些域名（尤其是国际邮箱）可能被 QQ SMTP 拒绝投递
- QQ 邮箱授权码可能过期（需定期在 QQ 邮箱设置中重新生成）
- 邮件可能进入收件人垃圾箱

## 方案 A（首选）：确认 QQ SMTP + 控制台降级

保持现有 QQ SMTP 配置，同时在服务端控制台输出验证码，当邮件发送失败或用户未收到时，服务端可直接查看验证码。

- 在 `sendEmailCode()` 中无论邮件发送是否成功，都将验证码打印到控制台
- 邮件发送失败时 API 仍返回错误，但用户可联系管理员从服务器日志获取验证码
- 优点：无需额外配置，改动最小
- 缺点：需要查看服务器控制台，不适合纯线上用户

## 方案 B（推荐）：阿里云邮件推送（企业级）

阿里云邮件推送（Direct Mail）是国内最稳定的邮件服务之一，支持每日 200 封免费额度，无需额外认证。配置方式：

- 注册阿里云 → 开通邮件推送 → 发信域名 → SMTP 密码
- 修改 `.env` 中的 MAIL_HOST/MAIL_USER/MAIL_PASS
- 稳定可靠，几乎不会丢信

## 方案 C（开发测试）：万能验证码模式

在开发/测试环境下，允许使用预定义的万能验证码（如 `666666`）直接登录，绕过邮件发送。仅限开发环境使用。

## 本次实现范围

采用 **方案 A**（首选）+ 同时写入 `.env.example` 方便后续切换方案 B。

**不采用方案 B/C 的理由**：方案 A 改动最小，先确认 QQ SMTP 实际是否能发送到非 QQ 邮箱；如果确认不可行再切换方案 B。

## 新增需求

### 需求：验证码控制台输出

服务器在发送（或尝试发送）验证码时，应在控制台打印验证码，方便开发和排障。

#### 场景：发送验证码
- **当** `sendEmailCode()` 被调用时
- **则** 无论发送成功或失败，控制台输出 `[验证码] 邮箱: xxx, 验证码: 123456`
- **则** 邮件发送成功时，控制台额外输出 `[邮件] 已发送至 xxx`

### 需求：配置模板

项目应包含 `.env.example` 文件，列出所有需要的环境变量及说明，方便新开发者配置。

#### 场景：项目初始化
- **当** 新开发者克隆项目后
- **则** 可参考 `.env.example` 创建自己的 `.env` 文件
